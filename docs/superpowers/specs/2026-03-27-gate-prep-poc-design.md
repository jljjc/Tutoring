# WA GATE & Scholarship Prep — PoC Design

**Date:** 2026-03-27
**Scope:** Proof of concept — small scale, end-to-end functional
**Target users:** Western Australia Year 6 students and their parents
**Target tests:** WA GATE (ASET) and private school scholarship tests

---

## 1. Overview

A web application that helps WA Year 6 students prepare for the GATE (ASET) test and private school scholarship tests. Students take randomised mock tests that mirror the real exam format and difficulty distribution. After each test, an AI tutoring engine explains each wrong answer in a personalised way — targeting the specific mistake the student made, not just the correct answer — and gives follow-up questions until the concept is mastered. Parents have a separate login to view reports, track progress, identify knowledge gaps, and receive actionable improvement suggestions. A projected GATE TSS score is updated after each test.

## 4. Test Formats

### 4.1 GATE / ASET Test
Mirrors the actual WA ASET format. Scored as Total Standard Score (TSS) out of 400.

| Section | Type | Questions | Time |
|---|---|---|---|
| Reading Comprehension | MCQ | 35 | 35 min |
| Communicating Ideas in Writing | Written response | 1 prompt | 25 min |
| Quantitative Reasoning | MCQ | 35 | 35 min |
| Abstract Reasoning | MCQ | 35 | 20 min |

**Total:** 105 MCQ + 1 writing task · ~2h 50min

### 4.2 Scholarship Test
Based on the Edutest format (most common WA private school scholarship provider).

| Section | Type | Questions | Time |
|---|---|---|---|
| English (Reading + Language) | MCQ | 30 | 30 min |
| Written Expression | Written response | 1 prompt | 20 min |
| Mathematics | MCQ | 30 | 30 min |
| General Ability / Reasoning | MCQ | 30 | 20 min |

**Total:** 90 MCQ + 1 writing task · ~1h 40min

### 4.3 Difficulty Distribution (MCQ sections)
Each MCQ section uses: **30% Easy · 50% Medium · 20% Hard** — matching the actual test difficulty curve.

Difficulty maps to the `question_bank.difficulty` integer scale as follows: **Easy = 1–2, Medium = 3–4, Hard = 5**.

## 5. Question Generation & Bank

Questions are AI-generated by Claude and cached in a shared `question_bank` table.

**Generation strategy:**
1. When a student starts a test, the system checks the question bank for enough *unused* questions for that student (questions they haven't seen before) across all required sections and difficulty levels.
2. If sufficient: pull from the bank and randomise order.
3. If insufficient: call Claude API to generate a batch for the needed section/topic/difficulty, cache the results, then serve.

This keeps Claude API costs low — questions are generated once and reused across different students. Each student gets a different randomised selection every session.

**Claude prompt structure for MCQ generation:**
- Test type, section, topic, difficulty level (1–5)
- Number of questions needed
- Output format: question text, 4 options (A–D), correct answer, brief explanation

**Writing prompts:**
- Generated fresh per session (not cached) to ensure variety
- Appropriate to Year 6 level and the test type (narrative/imaginative for GATE; persuasive/expository for scholarship)

## 6. Test Engine

### 6.1 Flow
1. Student selects test type (GATE or Scholarship) and starts
2. System assembles questions (from bank or generated fresh)
3. Sections delivered in order with a visible countdown timer per section
4. Student answers MCQ by selecting A/B/C/D; writing section provides a text input
5. Each answer saved to `test_answers` in real-time as the student progresses (prevents data loss)
6. On completion: score calculated, `test_sessions` record finalised, tutoring triggered

### 6.2 Anti-Repeat Guarantee
The seen-question set for a student is computed at test-assembly time by querying `test_answers JOIN test_sessions ON student_id`. No separate tracking table is needed for the PoC. A question is only served to a student who has no `test_answers` record for that `question_id`. When the unseen pool runs low, fresh questions are generated via Claude and cached in `question_bank`.

### 6.3 Practice Mode
For the PoC, students may run in **practice mode**: a single section (e.g. Quantitative Reasoning only) rather than the full test. Practice mode sessions:
- Generate the normal question count for that section only (e.g. 35 MCQ for Quantitative Reasoning)
- Trigger tutoring for wrong answers in the same way as full tests
- Are stored in `test_sessions` with `mode = 'practice'`
- **Do not contribute to the projected TSS score** (TSS projection requires a completed full test)

## 7. Writing Assessment

Both test types include one written response task per session.

**Delivery:** Student is shown a prompt and types their response in a timed text area.

**Scoring (via Claude):** Claude scores the response on 5 criteria, each 1–5:

| Criterion | What it measures |
|---|---|
| Ideas & Content | Relevance, depth, originality |
| Structure & Organisation | Introduction, paragraphing, conclusion |
| Vocabulary | Word choice, variety, precision |
| Grammar & Punctuation | Sentence construction, punctuation accuracy |
| Spelling | Accuracy of spelling |

**Output stored in `writing_responses`:** prompt, student response, scores per criterion (jsonb), AI feedback text, follow-up prompt.

**Tutoring for writing:** Claude provides specific written feedback identifying the weakest criterion, explains what to improve with a concrete example from the student's own text, then gives a new writing prompt targeting that criterion.

---

## 8. Personalised Tutoring Engine

Triggered automatically after test completion for every wrong MCQ answer. For writing, tutoring is triggered **per-criterion** only (any criterion scoring below 3/5). There is no aggregate writing score threshold.

### 8.1 MCQ Tutoring
Claude receives:
- The original question + all answer options
- The student's wrong answer (which specific option they chose)
- The correct answer
- Topic and difficulty level

Claude returns:
- Why the student's specific choice was wrong
- Why the correct answer is right
- A strategy or memory tip to avoid this mistake next time
- A follow-up MCQ question on the same concept

**Mastery loop:**
- Student attempts follow-up question
- Correct → topic marked as mastered
- Wrong → new explanation (different approach) + new question, up to 3 attempts
- After 3 failed attempts → topic flagged as a **Priority Gap** in the parent report

**Follow-up question sourcing:** Follow-up questions are generated ad-hoc by Claude at tutoring time and stored only in `tutoring_sessions.followup_question` (jsonb). They are **not** inserted into `question_bank` and do not receive a `question_id`. This means they are never served as main test questions and are not subject to anti-repeat tracking.

### 8.2 Writing Tutoring
- Triggered per-criterion: any criterion scoring below 3/5 triggers tutoring for the weakest criterion
- Claude gives specific feedback referencing the student's actual text
- Followed by a new writing prompt targeting that criterion
- Student submits new response; Claude re-scores and marks criterion as improved or still-developing
- Re-submissions and updated per-criterion scores are stored in `writing_tutoring_sessions` (see Section 9)

## 9. Data Model

```sql
-- Auth handled by Supabase; users table extends auth.users
-- email is intentionally omitted — read from auth.users when needed
users
  id            uuid PK (references auth.users)
  role          enum('student', 'parent')
  full_name     text
  created_at    timestamp

student_profiles
  id            uuid PK → users
  parent_id     uuid → users
  year_level    int default 6
  school        text
  target_schools text[]

question_bank
  id            uuid PK
  test_type     enum('gate', 'scholarship')
  section       text        -- e.g. 'reading_comprehension', 'quantitative_reasoning'
  topic         text        -- e.g. 'word analogies', 'fractions'
  difficulty    int         -- 1–2 easy, 3–4 medium, 5 hard
  question_text text
  options       jsonb       -- {A: '...', B: '...', C: '...', D: '...'}
  correct_answer text       -- 'A' | 'B' | 'C' | 'D'
  explanation   text
  generated_at  timestamp

test_sessions
  id            uuid PK
  student_id    uuid → users
  test_type     enum('gate', 'scholarship')
  mode          enum('full', 'practice')
  started_at    timestamp
  completed_at  timestamp
  total_score   int         -- total MCQ correct count
  section_scores jsonb      -- {reading: 28, quantitative: 30, abstract: 22, writing_total: 18}
                            -- writing_total is the raw sum of 5 criteria (0–25); normalised to 0–35 at TSS computation time
  projected_tss float       -- estimated GATE TSS out of 400 (full mode only)

test_answers
  id            uuid PK
  session_id    uuid → test_sessions
  question_id   uuid → question_bank
  selected_answer text
  is_correct    bool
  time_taken_secs int

writing_responses
  id            uuid PK
  session_id    uuid → test_sessions
  prompt        text
  response_text text
  scores        jsonb       -- {ideas: 3, structure: 4, vocabulary: 2, grammar: 5, spelling: 4}
  ai_feedback   text
  follow_up_prompt text

-- Stores writing tutoring re-submissions and updated scores per criterion.
-- writing_responses.scores is immutable (initial scores only).
-- The canonical current score for a criterion is the latest updated_scores row
-- by created_at for that criterion.
writing_tutoring_sessions
  id            uuid PK
  session_id    uuid → test_sessions
  writing_response_id uuid → writing_responses
  student_id    uuid → users
  criterion     text        -- e.g. 'vocabulary'
  follow_up_prompt text
  resubmission_text text
  updated_scores jsonb      -- per-criterion scores after re-submission
  improved      bool
  created_at    timestamp

tutoring_sessions
  id            uuid PK
  session_id    uuid → test_sessions
  student_id    uuid → users
  question_id   uuid → question_bank
  wrong_answer  text
  ai_explanation text
  followup_question jsonb   -- same shape as question_bank entry
  mastered      bool
  attempts      int default 0
```

## 10. Reporting

### Student view
- Test history (date, type, score, time taken)
- Section scores per test
- Progress over time (score trend per section)
- Knowledge gap map (topics colour-coded: red = priority, amber = developing, green = strong)
- Access to tutoring for any past wrong answer

### Parent view (all of the above plus)
- Projected GATE TSS score (out of 400), updated after each test
- Estimated percentile ranking vs GATE benchmarks
- **Improvement suggestions** — specific, actionable recommendations generated by Claude based on the student's weak topics (e.g. "Focus on pattern sequences this week — try 10 min daily. Here's what to practise...")
- Priority knowledge gap list (topics flagged after 3 failed tutoring attempts)
- 30-day progress summary

### Projected TSS Calculation (PoC)
Only calculated for **full** test sessions (not practice mode). A simple heuristic:
- MCQ sections: calculate % correct per section
- Writing: normalise the writing total (out of 25) to a 0–35 equivalent before weighting (i.e. `writing_score / 25 * 35`)
- Weight equally across the 4 sections, map to a 0–400 TSS range using approximate GATE score benchmarks
- Label with a percentile band (e.g. Top 10%, Top 15%, Top 25%)
- Stored in `test_sessions.projected_tss` and updated after every completed full test session

## 12. PoC Constraints & Simplifications

- **Single student per parent account** for the PoC (multi-child support is a future addition)
- **No payment/subscription** — open access for the PoC
- **No admin panel** — question bank seeded via script, not a UI
- **Writing prompts** are generated fresh per session (no caching needed for PoC)
- **Practice mode** allows single-section tests (no full 2h 50min commitment required)
- **Projected TSS** uses a simple heuristic benchmark, not official data
